# Концепция проекта: кастомный Matrix-клиент

## 1. Суть проекта

Разработка собственного мобильного мессенджера на основе протокола Matrix, реализованного как форк [FluffyChat](https://github.com/krille-chan/fluffychat) (v2.5.1, AGPL-3.0). Клиент сохраняет совместимость с экосистемой Matrix (Element, NeoChat и др.), но расширяется функциями, приближающими пользовательский опыт к Telegram.

## 2. Технологический стек

| Компонент | Технология |
|---|---|
| Язык / фреймворк | Dart / Flutter |
| Matrix SDK | `matrix` (Dart SDK от Famedly, v6.2+) |
| E2EE | `flutter_vodozemac` (Vodozemac, замена LibOlm) |
| WebRTC | `flutter_webrtc` + `webrtc_interface` (уже в зависимостях FluffyChat) |
| Видео | `video_player`, `video_compress`, `chewie`, `camera` |
| Навигация | `go_router` |
| State management | `provider` |
| Целевые платформы | Android, iOS (приоритет), далее — web, desktop |

## 3. Архитектура FluffyChat — текущее состояние

Кодовая база FluffyChat (~25–30 тыс. строк Dart без локализаций) организована следующим образом:

- **`lib/config/`** — конфигурация приложения (`app_config.dart`), ключи настроек (`setting_keys.dart`), маршруты (`routes.dart`), темы.
- **`lib/pages/`** — UI-логика, каждый экран в отдельной директории со своим контроллером и view. Ключевые: `chat/`, `chat_list/`, `dialer/`, `sign_in/`, `settings_3pid/`.
- **`lib/pages/chat/events/`** — виджеты отображения конкретных типов сообщений: `message_content.dart` (центральный switch по `messageType`), `video_player.dart`, `audio_player.dart`, `image_bubble.dart` и др.
- **`lib/utils/`** — утилиты, расширения SDK, `voip_plugin.dart` (VoIP-делегат).
- **`lib/widgets/`** — переиспользуемые компоненты; `matrix.dart` — корневой `MatrixState` (через `Provider`), владеет `Client`, `VoipPlugin`, `BackgroundPush`.

Ключевой паттерн: `MatrixState` → `Client` (Matrix Dart SDK) → вся бизнес-логика. UI-слой обращается к `Matrix.of(context)` для доступа к клиенту.

## 4. Перечень доработок

### 4.1. Видеозвонки 1:1

**Текущее состояние:** VoIP-инфраструктура в FluffyChat уже существует, но скрыта за экспериментальным флагом `experimentalVoip` (по умолчанию `false`). Имеются:
- `VoipPlugin` (`utils/voip_plugin.dart`) — реализация `WebRTCDelegate`, создаёт `VoIP(client, this)`, обрабатывает входящие/исходящие `CallSession`, управляет overlay.
- Страница звонка `pages/dialer/dialer.dart` (~610 строк) — показ видеопотоков, состояния вызова, кнопки mute/hold/screenshare, PIP-режим.
- Также есть отдельная Jitsi Meet интеграция (`jitsi_popup_button.dart`), открывающая конференцию в браузере.

**Что необходимо сделать:**

- Включить флаг `experimentalVoip`, провести аудит существующего кода на актуальность с текущей версией Matrix SDK.
- Добавить явную кнопку вызова (аудио / видео) в шапку 1:1 чата (сейчас доступна только Jitsi-кнопка).
- Стабилизировать сигналинг: обработка `m.call.invite`, `m.call.answer`, `m.call.candidates`, `m.call.hangup` через to-device события.
- Настроить TURN-сервер (coturn или аналог) для NAT traversal; добавить конфигурацию TURN в `app_config.dart`.
- **Android:** проверить интеграцию с `FlutterForegroundTask` для входящих звонков при заблокированном экране (каркас уже есть).
- **iOS (критично):** реализовать интеграцию с CallKit для VoIP Push Notifications. Без этого входящие звонки не будут приходить в фоне, а приложение не пройдёт ревью App Store. Потребуется нативный Swift-код в `ios/Runner/`.
- Edge cases: обработка прерываний (GSM-звонок поверх VoIP), переключение аудиоустройств (динамик / наушники / Bluetooth), wakelock, состояние сети.

**Точки входа в код:**
- `lib/utils/voip_plugin.dart` — основной делегат
- `lib/pages/dialer/dialer.dart` — UI звонка
- `lib/widgets/matrix.dart` — инициализация `voipPlugin`
- `lib/config/setting_keys.dart` — флаги `experimentalVoip`, `jitsiFeature`, `jitsiDomain`

### 4.2. Видеосообщения «кружочки»

**Текущее состояние:** в FluffyChat нет аналога video messages. Есть отправка видео как вложений (`MessageTypes.Video` → `EventVideoPlayer`) и запись голосовых сообщений (`RecordingInputRow` + `RecordingViewModel`).

**Протокольное решение:**

На уровне Matrix — это стандартное событие `m.room.message` с `msgtype: "m.video"`, но с дополнительным кастомным полем в `content`:

```json
{
  "msgtype": "m.video",
  "body": "video_message.mp4",
  "im.fluffy.video_message": true,
  "info": {
    "mimetype": "video/mp4",
    "duration": 12000,
    "w": 400,
    "h": 400
  },
  "url": "mxc://server/media_id"
}
```

Другие клиенты проигнорируют кастомное поле и покажут обычное видео (graceful degradation).

**Что необходимо сделать:**

- **Отображение:** новый виджет `CircleVideoMessage` — `ClipOval` вокруг видеоплеера, фиксированный размер (~200px), круговой индикатор прогресса, автоплей без звука, тап для включения звука. Точка ветвления — `message_content.dart`, `case MessageTypes.Video` (строка 182): проверка `im.fluffy.video_message` перед стандартным `EventVideoPlayer`.
- **Запись:** новая кнопка в `chat_input_row.dart` рядом с голосовыми сообщениями. По нажатию — экран записи с круглым превью фронтальной камеры. Пакет `camera` для захвата, ограничение 60 секунд.
- **Обработка:** сжатие видео перед отправкой (`video_compress`, уже в зависимостях), генерация thumbnail, кодирование в формат, совместимый с мобильными устройствами.
- **Отправка:** стандартный `room.sendFileEvent()` с дополнительным полем `im.fluffy.video_message: true`.

**Точки входа в код:**
- `lib/pages/chat/events/message_content.dart` — switch по типам сообщений
- `lib/pages/chat/events/video_player.dart` — текущий виджет видео (референс)
- `lib/pages/chat/chat_input_row.dart` — панель ввода
- `lib/pages/chat/recording_input_row.dart` + `recording_view_model.dart` — референс для аудиозаписи

### 4.3. Смена и восстановление пароля

**Текущее состояние:** в FluffyChat есть страница `settings_password/`, но она реализует только смену пароля при известном текущем пароле (через Client API `changePassword`). Нет сценария «забыл пароль».

**Что необходимо сделать:**

- **Смена пароля (авторизованный пользователь):** проверить и доработать существующий экран `settings_password`. Убедиться, что корректно работает UIA (User-Interactive Authentication) flow, включая 2FA если настроен.
- **Восстановление пароля (неавторизованный пользователь):** на экране логина добавить ссылку «Забыли пароль?». Реализовать flow через привязанный email: вызов `requestTokenToResetPasswordEmail` из Matrix Client API, ожидание подтверждения по ссылке, установка нового пароля через `setPassword`. Этот flow зависит от поддержки homeserver'ом и привязки email к аккаунту.
- **UX:** информативные сообщения об ошибках (email не привязан, homeserver не поддерживает, ссылка просрочена и т.д.).

**Точки входа в код:**
- `lib/pages/settings_password/` — текущий экран смены пароля
- `lib/pages/sign_in/` — экран логина, куда добавлять «Забыли пароль?»
- `lib/pages/settings_3pid/` — управление привязанными email (для контекста)

### 4.4. Улучшенный поиск пользователей

**Текущее состояние:** на странице `new_private_chat` есть поле поиска, вызывающее `searchUsers` из Matrix SDK. Однако поиск требует ввода полного Matrix ID (`@user:server.com`) или точного display name. Это неудобно для обычных пользователей.

**Что необходимо сделать:**

- **Поиск по подстроке display name:** при вводе текста в поле поиска — одновременно искать через `client.searchUserDirectory(term)` (уже поддерживает частичное совпадение по display name и user ID на уровне homeserver API `/user_directory/search`). Проверить, что homeserver корректно индексирует display names.
- **Нормализация ввода:** если пользователь ввёл что-то без `@` и `:` — не пытаться интерпретировать как Matrix ID, а сразу отправлять как поисковый запрос в user directory.
- **Поиск среди существующих контактов:** параллельно с серверным поиском — фильтровать локальный список уже известных комнат/пользователей по подстроке (мгновенный результат без сетевого запроса).
- **UX:** объединённый список результатов с приоритетом: сначала локальные контакты, затем результаты серверного поиска. Визуальное разделение секций.

**Точки входа в код:**
- `lib/pages/new_private_chat/new_private_chat.dart` — контроллер поиска
- `lib/pages/new_private_chat/new_private_chat_view.dart` — UI результатов

### 4.5. Привязка аккаунта Telegram (интеграция с мостом)

**Текущее состояние:** FluffyChat не имеет встроенной поддержки мостов (bridges). Для подключения Telegram через mautrix-telegram пользователь должен вручную найти бота моста (например, `@telegrambot:server.com`), создать с ним личный чат и вводить текстовые команды (`login`, затем номер телефона, затем код подтверждения). Это неудобно и неочевидно для обычных пользователей.

**Как работает mautrix-telegram:**

Мост mautrix-telegram — наиболее зрелая реализация моста Matrix↔Telegram. Он поддерживает два режима: puppeting (пользователь логинится своим Telegram-аккаунтом, и мост «представляет» его в Matrix) и relaybot (общий бот пересылает сообщения). Для puppeting-режима flow авторизации выглядит так:
1. Пользователь отправляет команду `login` боту моста.
2. Бот запрашивает номер телефона.
3. Пользователь вводит номер, Telegram присылает код подтверждения.
4. Пользователь отправляет код боту.
5. (Опционально) Бот запрашивает пароль 2FA, если он настроен.
6. После успешной авторизации мост начинает синхронизацию: создаёт Matrix-комнаты для Telegram-чатов и приглашает пользователя.

Альтернативный вариант — авторизация по QR-коду (команда `login-qr`): бот генерирует QR-код, пользователь сканирует его из официального Telegram-клиента.

**Что необходимо сделать:**

- **Новая страница в настройках:** `settings_telegram_bridge` — доступна из основного меню настроек. Содержит: статус подключения (привязан / не привязан), кнопку «Привязать Telegram», кнопку «Отвязать».
- **Конфигурация адреса бота моста:** в `app_config.dart` добавить настройку `telegramBridgeBot` (по умолчанию — `@telegrambot:<текущий_homeserver>`). Для кастомных серверов — возможность изменить в настройках.
- **UI flow привязки:** при нажатии «Привязать Telegram» клиент автоматически создаёт DM-комнату с ботом моста (если ещё не создана) и проводит пользователя через визард:
  - **Шаг 1:** Выбор способа входа — по номеру телефона или по QR-коду.
  - **Шаг 2 (телефон):** Поле ввода номера телефона с маской, отправка команды `login` и затем номера боту.
  - **Шаг 3 (телефон):** Поле ввода кода подтверждения из Telegram.
  - **Шаг 2 (QR):** Отправка команды `login-qr`, отображение полученного QR-кода для сканирования из Telegram.
  - **Шаг 4:** Поле ввода пароля 2FA (если бот запросил).
  - **Шаг 5:** Успешная привязка, информация о начале синхронизации чатов.
- **Определение статуса:** клиент проверяет наличие DM-комнаты с ботом моста и анализирует последние сообщения для определения текущего статуса авторизации. Альтернативно — команда `ping` боту, которая возвращает статус.
- **Отвязка:** отправка команды `logout` боту моста, подтверждение от пользователя, обновление статуса.
- **Обработка ошибок:** неверный код, истёкший код, неверный пароль 2FA, бот моста недоступен (мост не установлен на сервере), rate limit от Telegram.

**Технические особенности:**

Вся коммуникация с мостом идёт через обычные Matrix-сообщения в DM-комнате с ботом. Клиент не взаимодействует с Telegram API напрямую — только через бота. Это означает, что:
- Фича работает только если на homeserver'е установлен и настроен mautrix-telegram.
- Клиент парсит текстовые ответы бота для определения состояния (успех, запрос кода, ошибка). Формат ответов mautrix-telegram стабилен, но не является формальным API — потребуется аккуратный парсинг.
- Для QR-логина нужно извлечь изображение QR-кода из сообщения бота (мост отправляет его как `m.image`).

**Точки входа в код:**
- `lib/pages/settings/` — добавить пункт меню «Telegram»
- Создать `lib/pages/settings_telegram_bridge/` — новая страница с контроллером и view
- `lib/config/app_config.dart` — адрес бота моста по умолчанию
- `lib/config/routes.dart` — маршрут к новой странице

## 6. Зависимости и инфраструктура

- **Homeserver:** Synapse или Conduit с поддержкой VoIP (TURN-конфигурация в `.well-known`).
- **TURN-сервер:** coturn (самостоятельный или облачный), настроенный на homeserver'е.
- **Мост Telegram:** mautrix-telegram, установленный на homeserver'е как Application Service. Требует Telegram API ID и API Hash (получаются на https://my.telegram.org/apps).
- **Push-уведомления:** Firebase Cloud Messaging (Android), APNs + VoIP Push (iOS). В FluffyChat уже есть скрипт `add-firebase-messaging.sh` и поддержка UnifiedPush.
- **Apple Developer Account:** $99/год, обязателен для TestFlight и CallKit.
- **Google Play Console:** для закрытого тестирования на Android.
- **CI/CD:** GitHub Actions (FluffyChat уже использует), fastlane (настроен для iOS и Android).

- Нельзя менять лицензию на проприетарную.

## 8. Риски

| Риск | Вероятность | Влияние | Митигация |
|---|---|---|---|
| CallKit-интеграция на iOS окажется значительно сложнее ожиданий | Высокая | Высокое | Начать с Android-версии, iOS-звонки — только в foreground на первом этапе |
| VoIP-код в FluffyChat окажется сломан после обновлений SDK | Средняя | Среднее | Ранний аудит VoIP-кода (веха 6), фиксация версии SDK |
| TURN-сервер добавляет операционную нагрузку | Средняя | Низкое | Использовать публичный TURN (matrix.org) для тестирования, свой — для продакшена |
| Кастомное поле `im.fluffy.video_message` не станет стандартом | Низкая | Низкое | Graceful degradation: другие клиенты показывают обычное видео |
| Обновления upstream FluffyChat ломают кастомный код | Средняя | Среднее | Минимизировать инвазивные изменения, документировать все точки модификации |
| Мост mautrix-telegram не установлен на homeserver'е пользователя | Средняя | Среднее | Graceful degradation: если бот моста не найден — показывать сообщение о необходимости настройки на сервере, скрывать пункт меню |
| Формат ответов бота mautrix-telegram изменится в новой версии | Низкая | Среднее | Парсинг ответов с fallback на отображение «сырых» сообщений бота; версионирование парсера |
