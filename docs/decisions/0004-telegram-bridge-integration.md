# ADR-0004: Telegram Bridge — UI Over mautrix-telegram Bot

## Статус
Accepted

## Контекст
Нужна интеграция с Telegram для корпоративных пользователей, которые используют оба мессенджера. Варианты:

1. **Прямой Telegram API** — полный контроль, но: нужен Telegram API ID/Hash, поддержка MTProto, дублирование работы mautrix-telegram, два отдельных канала сообщений
2. **UI-обёртка над mautrix-telegram ботом** — бот уже реализует всю логику моста, клиент только предоставляет удобный UI
3. **Ничего не делать** — пользователи общаются с ботом вручную через текстовые команды

## Решение
UI-обёртка над mautrix-telegram ботом через Matrix DM-комнату.

Клиент:
- Автоматически находит/создаёт DM с ботом моста (`@telegrambot:<homeserver>`)
- Предоставляет визард авторизации (номер телефона → код → 2FA)
- Поддерживает альтернативный QR-login
- Парсит текстовые ответы бота для определения состояния
- Показывает статус привязки в настройках

## Последствия

### Плюсы
- Вся сложная логика моста (синхронизация чатов, пересылка медиа, puppet accounts) — на стороне mautrix-telegram
- Клиент не взаимодействует с Telegram API напрямую
- Работает с любым homeserver'ом, где установлен mautrix-telegram
- Не нужен отдельный Telegram API ID для клиента

### Минусы
- **Зависимость от формата ответов бота** — парсинг текстовых сообщений, а не формальный API. Формат стабилен, но не гарантирован.
- Работает **только** если на homeserver'е установлен mautrix-telegram
- UX ограничен возможностями бота (нет real-time status, нет прогресса синхронизации)

### Fallback при отсутствии моста
Если бот моста не найден на homeserver'е:
- Пункт "Telegram" в настройках показывает сообщение о необходимости установки моста администратором
- Функция не скрывается полностью (чтобы пользователь знал о возможности)

### Парсинг ответов бота
- Версионирование парсера для разных версий mautrix-telegram
- Fallback: если ответ не распознан — показывать «сырое» сообщение бота
- QR-код: извлечение из `m.image` события в DM-комнате
